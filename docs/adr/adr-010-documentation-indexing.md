# ADR-010: Documentation Indexing

*Status*: Accepted · *Date*: 2026-04-10 · *Context*: As the repo grows, finding the right document becomes harder for both humans and AI. A central index is needed. This approach is dogfooded from the personal repo's indexing strategy.

## Decision
Every markdown file gets frontmatter with a summary and tags. A central `INDEX.md` is generated from this frontmatter.

### Frontmatter format

```markdown
---
summary: One-line description of this document
tags: [tag1, tag2, tag3]
---
```

| Field | Required | Description |
|---|---|---|
| `summary` | Yes | 1 sentence, max 120 chars |
| `tags` | Yes | 3-5 searchable keywords, lowercase |

### INDEX.md
A generated file in the repo root listing all documents with their path and summary. Serves as a table of contents for both humans and AI.

### Generation
- Frontmatter is added manually (or via Ollama for bulk)
- `INDEX.md` is generated by a script that reads frontmatter from all `.md` files
- Regenerated before each release

## Rationale
- AI tools can read `INDEX.md` to understand the entire repo without scanning every file
- Humans get a searchable table of contents
- Frontmatter is a standard pattern (used by Jekyll, Hugo, Obsidian)
- Dogfooding: this approach is used and proven in the personal knowledge repo

## Consequences
- All documentation files should have frontmatter
- A `scripts/build-index.sh` script is needed (future)
- `INDEX.md` should be in `.gitignore` if generated, or committed if curated
