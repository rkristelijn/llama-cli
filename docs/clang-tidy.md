# Clang-Tidy Guide

`clang-tidy` is a C++ "linter" tool that helps catch typical programming errors, style violations, and bugs via static analysis.

## Usage in llama-cli

The easiest way to run `clang-tidy` is via the `Makefile`:

```bash
make tidy    # Run clang-tidy on all source files
make check   # Run all quality checks, including tidy
```

## Configuration

The project uses a `.config/.clang-tidy` configuration file which defines the enabled checks and warnings-as-errors policy.

## Suppressing Warnings

If a warning is a false positive or the code is intentionally written that way, you can suppress it using `NOLINT` comments. **Always include a comment explaining why the suppression is necessary.**

### Single Line Suppression
Use `// NOLINT` at the end of the line.

```cpp
// Suppress all warnings for this line
void* p = (void*)0x1234; // NOLINT

// Suppress specific check
void* p = (void*)0x1234; // NOLINT(cppcoreguidelines-pro-type-reinterpret-cast)
```

### Next Line Suppression
Use `// NOLINTNEXTLINE` on the line before the warning.

```cpp
// NOLINTNEXTLINE(clang-analyzer-valist.Uninitialized)
vfprintf(stderr, fmt, args); 
```

### Block Suppression
Use `NOLINTBEGIN` and `NOLINTEND` to wrap a block of code.

```cpp
// NOLINTBEGIN(cppcoreguidelines-avoid-magic-numbers)
const int SECRET_PORT = 1234;
const int BUFFER_SIZE = 512;
// NOLINTEND(cppcoreguidelines-avoid-magic-numbers)
```

## Best Practices

1. **Fix before suppressing**: Only suppress if fixing the code would make it less readable or if it's a confirmed false positive.
2. **Be specific**: Prefer `NOLINT(check-name)` over a generic `NOLINT` to avoid accidentally hiding other issues.
3. **Document reasons**: Future maintainers (and the AI) need to know why a check was disabled.

## Parallel Execution

For large changes, you can run `clang-tidy` in parallel if `run-clang-tidy` is installed on your system:

```bash
run-clang-tidy -p build/
```
*(Requires `compile_commands.json` generated by `cmake -DCMAKE_EXPORT_COMPILE_COMMANDS=ON`)*
